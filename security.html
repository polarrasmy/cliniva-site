بيانات مرضاك في أمان — بالتصميم

Cliniva مبني لقطاع الرعاية الصحية: كل قرار معماري يبدأ من سؤال واحد — «هل ده يحمي بيانات المريض؟». دي نظرة عامة شفّافة على كيفية حمايتنا لبياناتك وبيانات عملائك.

🇸🇦 توافق مع PDPL السعودي 🔒 تشفير شامل 🗂️ سجل تدقيق كامل ☁️ استضافة داخل المنطقة
الوثيقة دي بتوصف المعمارية الأمنية والتزامات Cliniva تجاه عيادتك. إحنا بنؤمن إن الأمن مش feature بتتباع — ده أساس. كل بند تحت قابل للمراجعة والنقاش مع فريق تقنية المعلومات عندك قبل أي تشغيل تجريبي (pilot).
☁️

سيادة البيانات (Data Residency)

بيانات عيادتك ومرضاك بتتخزّن على بنية تحتية سحابية داخل المنطقة (الخليج/السعودية) — ما بتخرجش خارج النطاق الجغرافي المتفق عليه.

In-region cloud
🔐

التشفير (Encryption)

كل البيانات مشفّرة أثناء النقل (TLS 1.2+) وأثناء التخزين (AES-256). مفيش بيانات مريض بتتنقل أو بتتخزّن نصّ صريح.

TLS 1.2+ · AES-256
👤

التحكم في الوصول (RBAC)

صلاحيات مبنية على الدور — أقل صلاحية ممكنة لكل مستخدم. الاستقبال يشوف المواعيد، المدير يشوف المالية، والطبيب يشوف مرضاه فقط. تسجيل دخول بخطوتين (MFA).

Least-privilege · MFA
🗂️

سجل التدقيق (Audit Trail)

كل تغيير مُسجّل: مين عمل إيه وإمتى — حجز، تعديل، إلغاء، أو فتح ملف مريض. سجل غير قابل للحذف، متاح للمراجعة في أي وقت.

Immutable log
💾

النسخ الاحتياطي والتعافي

نسخ احتياطية يومية تلقائية + استرجاع لنقطة زمنية (Point-in-Time Recovery). أهداف تعافٍ واضحة: RPO ≤ ساعة، RTO ≤ 4 ساعات.

Daily backups · PITR
⚖️

التوافق مع PDPL

مصمّم على مبادئ نظام حماية البيانات الشخصية السعودي (PDPL): موافقة المريض، تقليل البيانات، حق المريض في الوصول والحذف.

PDPL-aligned
🛡️

الخصوصية بالتصميم

بنجمع أقل بيانات ممكنة لتشغيل الخدمة، وبنعزل بيانات كل عيادة عن التانية (Tenant Isolation). بيانات عيادة ما توصلش لعيادة تانية أبدًا.

Privacy by design
🚨
